# Working in this agent

This project defines exactly one Sapiom agent in `index.ts` — **Personalized Media at Scale** — authored against `@sapiom/agent`. It has five steps: `fetch` (reads recipient rows from `database`) → then either `renderImages` (calls `contentGeneration.images`) or the `renderClip` ⇄ `collectClip` loop (calls `contentGeneration.video`, async) → `deliver` (emails each recipient via `email`). Inside a step's `run`, Sapiom capabilities are pre-auth'd on `ctx.sapiom` (e.g. `ctx.sapiom.database.get(...)`, `ctx.sapiom.contentGeneration.images.create(...)`, `ctx.sapiom.email.messages.send(...)`).

The graph forks on the chosen `medium`: images are cheaper and made all at once; video is async and rendered one row at a time so a paused step is always waiting before its job can complete (the `scene-to-video` lesson).

## Authoring

- An agent is `defineAgent({ entry, steps })`; each step is `defineStep({ name, next, run })`. Keep exactly one `defineAgent(...)` export.
- **Capabilities come from the types.** What's available on `ctx.sapiom` is defined by `@sapiom/tools` — read the types / use autocomplete rather than guessing. A wrong capability or method name fails typecheck.
- **No LLM in the loop.** The per-row prompt is built deterministically from the row's `context` (`buildPrompt`). This template generates media, not copy. If you want the prompt itself written per row, add `ctx.sapiom.models.run` inside the render step and list `models.run` in the registry `capabilities`.
- **Keep the edges slim.** Only slim rows and asset references cross step boundaries; generated bytes live in file storage, referenced by `fileId` / `downloadUrl`, never in `ctx.shared`. Large shared state stalls transitions on the cloud engine.
- **Gate real side effects behind `dryRun`.** `fetch` stops after reading the rows and returns the planned prompts; nothing is generated or emailed. Keep new external side effects behind the same guard.
- **Fan-out vs. loop.** Images fan out with `Promise.all` (all at once). Video must go one clip at a time through `renderClip` → `pauseUntilSignal` → `collectClip` → back to `renderClip`; don't launch every clip up front or a job could finish before its step has paused.

## Validating

When you've made a coherent change and want to validate it — the same point you'd run tests in any project — reach for the local suite. You don't need to run it after every small edit.

- **`npm run typecheck`** — types, and confirms every `ctx.sapiom.*` capability/method you used exists.
- **check** — typecheck + bundle + manifest + step-graph validation. The full local pre-flight before deploy.
- **run_local** — runs your **real** step code against **stub capabilities**, so `database` / `contentGeneration.images` / `contentGeneration.video` / `email` return built-in defaults and the agent runs end-to-end offline for free. Pass `dryRun: true` so `fetch` returns the plan without generating or sending. Returns a per-step trace.
- **deploy**, then **run** — ship it, then perform a real, billed batch: images or clips generated, stored, and emailed. Attach the `schedule` as a cron trigger to run it on a cadence.

> Write each step the way it should run in production. `run_local` adapts to your code (stub capabilities), not the other way around — never weaken or drop real logic to shape a local run.

Drive `check` / `run_local` / `link` / `deploy` / `run` via the Sapiom MCP dev tools (`sapiom_dev_agents_*`). See `README.md` for the full lifecycle.

## Database

`fetch` reads from a Postgres table (`media_recipients`) inside the database handle `personalized-media`. It creates the database (`ctx.sapiom.database.create`) and seeds a few demo rows the first time so the template runs with no setup. Point `dbHandle` at your own table — it needs `name`, `email`, and `context` columns — once you're past the demo. The Postgres client (`postgres`) opens with `ssl: "require"` and is closed in a `finally`. Reading rows is free, so a `dryRun` still reads them (to build the preview) and only skips generation + email; under stubbed capabilities in `run_local` there's no connection string, so `rows` stays empty and the graph still traces.

## Determinism

A step body runs **once** on the happy path; it re-runs only on retry (after a throw). Capture non-deterministic values (timestamps, ids) once and pass them forward via the `goto(...)` input or `ctx.shared` rather than recomputing them.
